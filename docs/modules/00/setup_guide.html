CS 344 Setup Guide (eCampus)

In this course, you will complete your coursework on a linux server called os1, which is hosted on the OSU network.

  • os1 has a specially configured environment for this course, which is not available on other school servers or your local machine.

  • Students in this class often test code which can cause server instability; you are prohibited by IT from working on assignments in this class on other servers, such as flip. Violating this policy can result in suspension of your access to the OSU network!

Remote development work through a command-line interface is a fundamental skill in software development. You may be evaluated on your proficiency with the tools of the trade that we introduce in this class.

Warning

Connecting to os1 with the vscode remote SSH extension is prohibited. Aside from sidestepping course learning objectives, vscode server threatens the stability of os1 for other users. Students caught running instances of vscode server on os1 will face disciplinary action.

Pre-Setup

You may need to create an engineering account on T.E.A.C.H. in order to access the school servers. This is an older system that was used to manage user accounts before ONID. These accounts are now unified, but you’ll need to activate your T.E.A.C.H account if you haven’t already done so in a previous class.

After logging into T.E.A.C.H, please navigate to the “Change Unix Shell” resource under “Account Tools” on the right-side navigation menu. Ensure that your shell is set to bash. If it is not set to bash, change it, and then wait at least 15 minutes before continuing with the rest of the configuration, because it seems to take some time to propagate to the servers.

Configuration Process Overview

Configuring your environment to do software development on os1 has only a few major steps, but it is important to understand what you are doing and why in order to solve any problems you run into along the way. You will:

  • Install a modern terminal emulator application.

  • Install and configure necessary utilities on the local machine in order to connect to the os1 server.

  • Configure your OSU user account in order to access resources provided on os1.

Installing a Terminal Emulator

A terminal emulator is an application that emulates a physical terminal device (teletype keyboard and display). The terminal emulator provides a window that captures mouse and keyboard input to send to another terminal-based application, and which displays output from that application.

os1 was configured and tested with kitty[1], a fast, feature-rich, GPU based terminal emulator. I highly recommend using kitty, if you are able, in order to experience the course as intended. How you go about installing a terminal emulator depends on the operating system on your local machine.

Windows

Windows users will need to install Linux in order to run a functioning terminal emulator. You have three options:

  1. Install a Linux distribution on a separate partition. Some of the most popular and beginner friendly are:

    • Ubuntu

    • Arch

    • Debian

    • Gentoo

    • Fedora

    • Find the perfect distro for yourself at distrowatch

  2. Install Windows Subsystem for Linux with a GUI desktop environment: Win-Kex

  3. Install Cygwin

Option 1

Installing Linux as a native operating system is strongly recommended for the best user experience. Ubuntu is the most beginner friendly option.

After installing linux, continue to MacOS, Linux, BSD, etc.

Option 2

Installing Linux under WSL is recommended only if you are unable or unwilling to install a standalone Linux distribution on your personal system. The quality of WSL improves each term, but it is still fairly unstable and many students have found it practically unusable–switching to a full linux install in the middle of the term is not uncommon.

There are several pitfalls with WSL:

  • In order to run WSL, your system must support hardware virtualization. Several students have had to enable this option in their motherboard’s firmware (BIOS/UEFI).

  • You cannot simultaneously use the OSU VPN while also using WSL to access the internet, due to a #wontfix bug in WSL.

  • Moving files between WSL and Windows can be confusing. Submitting assignments can be frustrating

  • Some students have found that WSL does not support graphics acceleration for their particular GPUs, resulting in black screens, random crashes, and other display issues. Often students end up having to use outdated terminal emulators as a result.

After installing WSL with Win-Kex continue to MacOS, Linux, BSD, etc.

Option 3

Installing Cygwin is recommended as a last resort because it is horrendously unpleasant.

  • Cygwin is extremely unstable and crashes constantly.

  • There is no uninstaller for Cygwin. The many files it installs must be manually removed. Some people consider it malware for this reason alone.

  • Cygwin has no package manager. Installing additional software packages requires completely reinstalling Cygwin.

  • Cygwin does not support graphics acceleration, forcing students to use outdated terminal emulators.

  • Cygwin uses noncompliant versions of standard utilities that often do not work properly.

The default Cygwin terminal emulator, Mintty, is the best you will get with Cygwin, and it does not work very well.

MacOS, Linux, BSD, etc.

Install kitty[1] terminal emulator. If, for some reason, kitty does not work properly (such as if your system does not support GPU acceleration), the following terminal emulators are suggested as acceptable alternatives:

  • Alacritty

  • Gnome Terminal

  • Foot

  • iTerm2

The following terminal emulators are known to have major issues:

  • Terminal.app (MacOS)

  • Terminator

  • ConEmu

  • MobaXterm

  • PuTTY

  • CMD.exe

  • Windows Terminal

  • Xterm

[1] (1,2)

https://sw.kovidgoyal.net/kitty/

Local Setup

In order to connect to the remote server, you will need an SSH client, ssh. It is also recommended that you install a text editor (vim or nvim), and git.

These should all be available through the appropriate package manager for your system;

Debian/Ubuntu/Kali
sudo apt-get install -y kitty openssh-client neovim git
Arch
pacman -Syu kitty openssh neovim git
Gentoo
emerge --ask x11-terms/kitty net-misc/openssh app-editors/neovim dev-vcs/git
Brew
brew install kitty openssh neovim git

Next, you need to set up your ssh client to connect to os1,

  1. Generate public/private key pair

  2. Install keys to OSU network account

  3. Create ssh config

Generate Public/Private Key Pair

You will connect to OSU servers using public key cryptography. The way this works is you generate a pair of keys which are symmetric: a message encrypted with one key can be decrypted with the other. One of these keys will be a private (secret) key, and the other public (published). You confirm your identity by decrypting a challenge message that was encrypted with your public key; this has several major advantages:

  • Public key cryptography is convenient, allowing passwordless logins.

  • Your private key is never exposed, allowing it to be used everywhere, unlike a password.

  • Each local device has its own private key. If a device is compromised, that credential can be revoked without having to update a master password on every other device.

Generate a keypair with the following command:

ssh-keygen -t ed25519

This will ask you a series of questions, and then generate your two key files.

If you choose to enter a password, the password will be used to encrypt your private key file. This is unrelated to your ONID password. The password will be required to be entered each time the private key file is used to authenticate; you can later change or remove this password. The risk of not having a password is that the key file is stored unencrypted on the disk, where an attacker with physical access or remote administrative access could retrieve it. For most students, this is an unnecessary level of security. If you use a password, you may want to install an ssh-agent service which stores your decrypted private keys in memory to be reused on subsequent connections without requiring password reentry; this is beyond the scope of this guide.

If you chose the default file locations, ~/.ssh/id_ed25519 is your private key and ~/.ssh/id_ed25519.pub is your public key. The private key is a secret. Do not share it with anyone.

Register Public Key Credential with Server

Next you need to log into a school server (using your ONID password and DUO), and copy your public key to the authorized_keys file on the server. The following command automates this and ensures that the files on the remote system are created with appropriate permissions to work properly:

ssh-copy-id your-onid-username@access.engr.oregonstate.edu

Notice that your-onid-username should be replaced with your onid username.

You should be prompted for your ONID password. Then it will ask you to authenticate with DUO. If you have a phone registered, you should be able to send a push notification with 1, or you can directly enter a 6-digit temporary PIN generated by your hardware token or DUO app.

ssh-copy-id will then copy your public key(s) to the remote server and attempt to log in with them. You may be asked to enter a password to decrypt your private key file if you entered a password during the key creation step.

Confirm Key Registration

You should now be able to log in without entering your ONID password and using DUO:

ssh your-onid-username@access.engr.oregonstate.edu

Notice that your-onid-username should be replaced with your onid username.

As before, you may need to enter in a password to decrypt your private key file, if you entered one during key generation.

Note

If you see a message about an unknown terminal “xterm-kitty”, logout and reconnect with:

kitty +kitten ssh your-onid-username@access.engr.oregonstate.edu

This will install the terminfo for kitty to your user account. This will be likely be necessary until the servers are updated.

Configure SSH Client

You have successfully installed your keys on flip which is the bastion server accessed via access.engr.oregonstate.edu. The os1 server is not public-facing and can only be accessed from within the OSU network. If you are on the VPN or physically connected to OSU’s network, you may be able to access os1 directly.

Important

You are creating a file on your local machine to configure your local ssh client. If you are still logged into flip from the previous step, log out (logout). Confirm that you are not logged into a school server with the hostname command before proceeding:

$ hostname
flip2.engr.oregonstate.edu
$ logout
$ hostname
bennyb-laptop

Edit/create ~/.ssh/config to have the following entries:

Host *
   ServerAliveInterval 10
   ServerAliveCountMax 6

Host flip os1
  User your-onid-username
  LogLevel=error

Host flip
  Hostname access.engr.oregonstate.edu

Host os1
  ProxyJump flip

You can use a graphical text editor to do this, if you are uncomfortable using something like vim or nano in the terminal.

You should now be able to connect directly to os1 without entering your ONID password,

$ ssh os1
$ hostname
os1.engr.oregonstate.edu
