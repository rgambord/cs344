CS 374 - Operating Systems
  • Introduction

Introduction

This course covers a lot of materials and for many is their first approach to C programming and related OS concepts. Before we begin, let’s introduce ourselves to the major resources and tools we will be using in the course. Next we will review the general course structure and design philosophy so that you can better understand how your instructor intends for you to engage with the course. In the remaing pages of this module, we will take the Syllabus Quiz, walk through the Software Setup Guide, and learn to use some of the software tools we will be expected to be familiar with in the first week of the course.

Major Resources and Reference Works

In this class, we will study several resources in order to learn more about the broad topic of Operating Systems. A large amount of our readings will involve direct analysis of standards documents and reference materials.

Standards

  • Single Unix Specification – The Single UNIX Specification, version 4 (SuSv4), published by the Open Group, a global consortium dedicated to open, vendor-neutral technology standards. SuSv4 defines the minimum standard that an operating system must meet in order to be able to be called a “UNIX” operating system. SuSv4 is the collation of three major works:

    • C Language – The de facto language of systems programming since its debut the early 1970s. SuSv4 aligns with and extends, but does not directly incorporate, the 1999 version of this standard: ISO/IEC 9899:1999, also called C99.

    • POSIX – The Portable Operating System Interface standard defines, broadly, the basic features of a POSIX operating system, the system interfaces and a set of standard system utilities. SuSv4 is largely based on the 2008 version of this standard: ISO/IEC 9945:2009, also published as IEEE 1003.1-2008, often called POSIX.1-2008. POSIX is also published as a standalone specification by the Open Group.

    • XCURSES – X/Open Curses describes a set of interfaces for terminal-independent methods of updating character screens. X/Open was an organization that combined with the Open Software Foundation in 1996 to form the Open Group. XCURSES is directly defined as a volume of the SuS, and is not a standalone specification.

  • System V – System V Application Binary Interface, edition 4.1 (SRV4), published in 1996. System V (“System Five”) was one of the first commercial versions of the Unix Operating System developed by AT&T. SYSV, and its Executable and Linkable Format (ELF) remains the most widely adopted ABI and executable file format specification among all Unix-like systems.

Local Documentation

Throughout the course, we will routinely refer to local documentation on our class server as we complete our assignments. The software utilities and system interfaces actually installed on a system are implementations of the standards mentioned above, but may (often) have additional features or deviations from the standards. Local documentation describes these actual implementation details and should be your first-line reference when working on a particular system.

Beware! – Local documentation is version-specific to the software on the system, it’s important to be mindful that newer online versions of these resources (such as man7.org) may have subtle, but critical, differences, which can lead to a lot of head scratching!

  • Manual pages – Every system comes with a set of “man” pages; originally distributed with operating systems as actual published manuals, man pages are now distributed in a digital format, and are specific to the versions of software present on the systems on which they are found. We will use the man system utility to navigate the digital manual.

  • Info pages – GNU, the open source project that implemented many of the system utilities used by different Unix-like operating systems, also includes the GNU info utility. Similar to man pages, Info pages go into greater depth and detail about a particular topic. While man pages are generally intended as a quick reference, info pages are meant to be more of a learning material.

Textbooks

Both textbooks for this course are available for free online.

  • The Linux Programming Interface – TLPI organizes and explains the use of the Linux System Interfaces from the perspective of the needs of a system programmer. While the standards generally describe abstract behaviors and requirements, TLPI covers the implementation details of specific interfaces in the actual Linux operating system.

  • Modern C – Modern C is a guide to C programming written by Jens Gustedt. Jens is a renowned expert in C programming, and sits on the C standards committee, JTC1/SC22/WG15.

Major Software Tools

We will use several key pieces of software to accomplish our assignments and learn more about operating systems. All of this software is pre-installed on our class server and ready for you to use. With too many to list here, some of the most important to be aware of are:

  • *nix – “*nix” is shorthand for “Unix-like Operating System”. As we will learn later, this mostly refers to three major branches within the Unix family tree:

    • Linux/GNU – the free and open source operating system, created by Linus Torvalds in 1991

    • Darwin/MacOS – Apple’s operating system, first created in the early 2000s

    • BSD – The descendants of the other free and open source Berkley Software Distribution from 1978

    In this class, we will do all of our work on a Linux server hosted at OSU, called os1.

  • System Utilities – There are 160 standard POSIX system utilities, and we will learn to use a subset of these for common tasks. The most important are:

    • Shell – The command-line shell provides the primary mechanism for interactive control of the operating system: the command-line interface. There are several popular implementations of the shell specification; in this class, we use Bash (Bourne Again Shell).

    • Vim – The industry-standard terminal-based text editor, widely used by systems programmers for editing files and working on software projects. Vim is often the only text editor available on headless, remote, or embedded systems, and professional programers need to have a basic level of familiarity with it. In this class, we use a modern reimplementation called NeoVim that has been heavily preconfigured for this class.

    • C compiler – The C compiler translates source code into executables. In this class, we use the gcc (GNU Compiler Collection) compiler implementation and write our code according to the C99 language specification.

    • Make – Part of the Software Development Utilities (SDU) of POSIX, Make is used to maintain, update, and regenerate groups of programs. We will use Make to build our projects, specifically, the GNU implementation, which has many additional features (gmake).

  • GNU Toolchain – A collection of programming utilities distributed by the GNU project. GCC and GNU Make were mentioned above. Other software included with the GNU Toolchain are:

    • GNU C Library (glibc) – An implementation of the C standard library.

    • GNU Binutils – Various tools for linking and assembling binary executables.

    • GNU Debugger (GDB) – A powerful code debugging tool.

    • GNU Autotools – A suite of tools for packaging software for source-based distribution; one of the most widely used packaging formats, with its signature 3-step installation command:

      sh ./configure && make && make install

  • Doxygen – A powerful tool that generates documentation through familiar docstring (/**) comments and static code analysis.

Course Design and Philosophy

Operating Systems is a notoriously challenging topic for many students. It’s very common for students to feel lost and overwhelemed by the sheer volume of information that is available to them. This course is designed to introduce you to several key OS concepts while staying within a walled-garden of information that allows you to feel grounded by fully understanding what you must know before starting an assignment.

How to approach readings

Software developers have to read a lot of documentatin. In this course, we will practice effective strategies for approaching the sometimes overwhelming amount of information present in documentation materials. We will also familiarizing ourselves with the language and writing style that most documentation uses.

Each module has several assigned readings, and these readings are critical to your success in the course. Expect to read (or review) 100-200 pages of materials per week. Often, you will be directed to skim a large number of pages of reference materials in order to understand the high-level structure and key information presented, so that you are primed to return to those materials later when questions arise, or when asked to review a particular section.

Communication

With so many different ways to communicate online, keeping track of all the possible communication channels can be overwhelming. For this reason, we do not use any canvas features to communicate directly with students, such as Canvas Announcements, Canvas Inbox, or assignment comments. We communicate via email for personal matters (grading, illness, extensions, etc), and via Ed Discussions for all other course-related information. You are expected to check your email and Ed at least once per business day. You are expected to follow along with Ed discussions that take place, just as you would pay attention to in-class discussions in a traditional classroom setting.

Office hours are held via Microsoft Teams, according to a public google calendar that the TAs are tasked with keeping up to date. TAs are required to make an announcement on Ed when making changes to their scheduled office hours that fall within the next week; otherwise they are free to make silent changes to their office hours for dates that are further out. Microsoft Teams is not used for any other purpose unless agreed upon in advance (such as a meeting previously scheduled via email).
