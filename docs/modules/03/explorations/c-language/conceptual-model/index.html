Conceptual Model

The basic conceptual model of C describes the context of a C program, from its beginning as a collection of source files through to the end of its execution. This model defines the requirements that an external environment must satisfy in order to build and execute a C program. It also describes the expected behavior of a C program from this external perspective; i.e. the input and output characteristics of a program that is written correctly and complies with the standards.

Environment

The implementation is the (set of) software that facilitates translation of C source code into an executable format, and the subsequent execution of functions in the translated executable. The contexts in which these two tasks are accomplished are called the translation environment and the execution environment.

In many cases, a single system encompasses both of these environments, but in others they may be separated and further divided across two or more systems, such as when building software on a development machine (translation environment) that is meant to be deployed on a separate embedded system (execution environment) that lacks its own facilities to translate code into executables. The distinction between the translation and execution environments is one of the reasons C is such a versatile language – many other high level languages like Python and Java require runtime support, such as an interpreter, which may be infeasible to deploy onto a resource constrained target system such as an embedded device.

The Translation Environment

The translation environment is the portion of the implementation that supports conversion of C source code into an executable that can be run on the target system (the execution environment). Translation begins with a C source file and any other associated source files, libraries, and header files, which are collectively called a preprocessing translation unit. The eight phases of translation can be grouped into three main stages: preprocessing, compilation, and linking.

Preprocessing

The first six translation phases involve various text-processing steps that operate on the preprocessing translation unit in a purely syntactic manner, such as replacing comments with spaces, concatenating adjacent string literals, and performing pre-processor macro expansions. The output of this stage is called the translation unit and is conceptually a list of preprocessor tokens – syntactic tokens classified as identifiers, punctuation, numbers, and so on.

Compilation

In the seventh phase, the syntactic tokens are further reinterpreted semantically, and compilation takes place, producing a compiled translation unit. The output of this stage is native machine bytecode, which is also known as object code. Object code is simply a collection of the pieces of a program (binary data, and functions as bytecode), along with a partially complete symbol table that points to each function or object, but lacks the “glue” that binds, or links, all of the pieces into an actual standalone executable.

Linking

In the final phase, compiled translation units and library components needed to satisfy external references (i.e. complete the missing references in the symbol table, which correspond to external library components) are collected to form a program image that contains all the information needed for execution in its execution environment. In other words, this produces an executable, or loadable, program from the pieces contained in object files that compilation produced.

These three stages are often carried out in a single step, and translation is often casually referred to simply as compilation. In fact, POSIX (The standard which comprises the bulk of UNIX) specifies a single program, c99, the C compiler, which performs all of these actions, but which can also be directed to halt at specific points.

For example, translation can be halted before linking, producing standalone object code. The object code cannot be executed directly, but it is ready to be linked without needing to be recompiled. This is how large projects are built; each individual source file is independently built into an object file, and then all of the object files are linked in a separate step to produce a final program. The benefit of this is that changes to a single source file don’t require re-compilation of the entire project. Only that single file needs to be rebuilt, and then the whole project can be re-linked (which is relatively fast). This is also how libraries work – the library is built once and then its object code is simply linked against without needing to be rebuilt for every project that uses it.

In this course, we will be using gcc (GNU[1] Compiler Collection) to carry out the stages of translation. Another popular alternative is the clang compiler frontend which transpiles C source into an abstract syntax that it then passes to the separate language-agnostic backend, LLVM, for actual compilation. Of the two, gcc is much older, and has a lot of inertia behind its adoption, but clang has recently caught up to, and now rivals the performance of gcc in many aspects. The choice of which to use is largely personal, and they both support many of the same features and command-line arguments, but since we will also be studying the entire GNU build system, we will stick with gcc for this course.

The Execution Environment

The execution environment is the system on which the compiled program is actually run. This more or less refers to the operating system (or lack thereof), and any additional support utilities (such as a loader/dynamic linker) or other running programs that affect the execution of a program. The most fundamental property of the execution environment is whether it is hosted or freestanding.

A hosted environment provides an operating system along with a full implementation of the C standard library. A hosted environment manages program startup by calling into the standardized entry point of the program, a function called main, which is passed a list of command-line arguments. The hosted environment also supports program termination (either by returning from main, or by calling an appropriate exit function), and allows the program to pass back a return, or exit, value to the environment.

On the other hand, a freestanding environment provides only a minimumal subset of the standard library, and the entry point of a program is defined by the implementation. This allows for C programs to be written for a wide variety of systems, including embedded systems, where an operating system is not present. In fact, operating systems, themselves, are written for, and execute within, the context of a freestanding environment.

Conformance

A C program which uses only the language and library features described in the standard is said to be strictly conforming, and its behavior is well-defined. If a hosted or freestanding implementation can accept a strictly conforming program, the implementation is said to be, itself, conforming. Finally, a program which is not strictly conforming, but which is acceptable to one of these conforming implementations and does not violate the standards is called a conforming program.

Strictly conforming programs are limited in what they are capable of doing, since they are restricted to only the basic features of the standard library, but their advantage is that they are portable across all conforming implementations. Conforming programs, on the other hand, often depend on non-portable features of the implementation, such as language extensions and additional libraries, or rely on system interfaces of a particular operating system, which reduces the portability of the program while dramatically increasing its capabilities.

Note

Later in this course, we will also be studying the POSIX (Portable Operating System Interface) standards which describe the fundamental features of a UNIX-like operating system, including system interfaces, filesystem structure, and additional system libraries that extend beyond the standard C libraries. The majority of the programs we will write will not be strictly conforming programs, and will instead use these additional features. As a result, our programs can be expected to be portable across all POSIX-compliant systems, which, for all practical purposes, is everything that is not Windows.

The C Abstract Machine Model

The C abstract model is a conceptual model that describes how a C program is executed. It consists of a set of rules and assumptions that define the behavior of C programs without specifying how they should be implemented on a particular computer or operating system, or concerning itself with optimization or performance.

According to the abstract model, accessing a volatile[2] object, modifying an object or file, or calling a function that does so, is called a side effect because it changes the state of the execution environment. The program is broken up into a series of sequence points, each of which marks a stable, settled, state in the program where all prior side effects have been resolved, and all following side effects have not yet occurred. Otherwise, the order of side effects that occur between sequence points is indeterminate.

From the external perspective, the main requirements of a program’s execution are that:

  • At each sequence point, volatile objects are stable and any accesses have completely occurred, or have not yet begun.

  • At program termination, all data written to files must be identical to what would be exepected if the program executed exactly as described in the standards.

  • The dynamic input and output characteristics of the system are as described in the standard. For example, it should be possible to ensure that prompt messages appear on an interactive terminal before a program waits for input, even though input and output are accessed separately.

At this high level, an entire C program can be conceptualized as simply a sequence of state-transitions which take place in the execution environment, according to the source code interpreted by the semantics of the language. The visible effects of this are called the program’s behavior.

Behavior

A strictly conforming program always exhibits well-defined, or specified, behavior on execution. However, as mentioned above, strictly conforming programs are very limited, so most useful programs are not strictly conforming. Such programs exhibit behaviors which may be categorized as unspecified, implementation-defined, locale-specific. Finally, undefined behaviors are ones which no conforming C program may ever invoke, as the results are completely indeterminate.

Unspecified Behavior

In many situations, the standard provides for more than one possible behavior, but places no further restrictions on which behavior is chosen in any instance. This is called unspecified behavior. An example of this is that the individual arguments to a function call are each evaluated, but the order in which they are evaluated is unspecified, so that if one argument is an expression which has some side effect when evaluated, it’s indeterminate whether that side effect will have happened before, or after, another argument is evaluated.

Implementation-defined Behavior

Implementation-defined behavior is like unspecified behavior, except that the implementation is required to choose a consistent choice between several possible behaviors and document that choice. For example, the result of many operations involving signed integers, such as out-of-range results, are implementation-defined behavior.

Locale-specific Behavior

Locale-specific behavior is that which depends on the specific localization of the execution environment, taking into account aspects of nationality, culture, and language. For example, many of the library functions used to process or categorize text exhibit locale-specific behaviors.

Undefined Behavior

Undefined behavior (UB) is the boogeyman of C-programming, and is a fairly unique aspect of the language that often frustrates and amazes learners. UB is a catch-all category for everything that doesn’t fall into one of the above categories, and invoking undefined behavior can have any result at all, famously referred to as nasal demons.

Undefined behavior exists for two very important reasons. First, because there are few restrictions and safety mechanisms in place, it’s much easier to implement the C language than it is other safer languages. This is the primary reason why C became as ubiquitous and enduring as it is, because it was so simple to port a C compiler to different systems, at a time when every system ran proprietary hardware and operating systems that were all mutually incompatible.

The second reason UB is so powerful is because it enables incredible amounts of optimizations. The compiler is free to assume that UB does not exist – that the programmer never makes such a mistake, and thus can optimize in extremely clever ways. Without UB, compilers would have a much more difficult time ensuring programs run quickly while also remaining correct.
